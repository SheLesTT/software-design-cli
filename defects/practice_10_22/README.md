## 1. [Диаграмма классов](./ClassesDiagram.png)
![Диаграмма классов](./ClassesDiagram.png)

**DefectSystem** - главный класс, отвечает за хранение и работу всей системы.

**Human** - базовый класс с информацией о сотрудниках предприятия.

**Worker** - класс рабочего, хранит всю информацию о рабочем, позволяет работать с его занятостью.

**Master** - класс бригадира.

**Team** - класс бригады, хранит информацию о членах бригады, бригадире и позволяет узнать занятость бригады и результаты ее работы.

**RepairZone** - класс зоны ремонта, хранит информацию обо всех ремонтных местах и текущей бригады в ней.

**RepairPlace** - класс ремонтного места.

**Shift** - класс рабочей смены, хранит информацию о времени смены, том какие бригады в каких зонах, а так же обо всех дефектах и ремонтах за смену.

**Repair** - класс ремонта, хранит информацию о времени ремонта, где был производен ремонт, кем был произведен ремонт, и какой дефект исправляли.

**Defect** - класс дефекта, хранит всю необходимую информацию об автомобиле и месте дефекта, а также где, когда и кем был диагностирован дефект. Позволяет начать ремонт этого дефекта.

## 2. [Диаграмма компонентов](./ComponentDiagram.png)
![Диаграмма компонентов](./ComponentDiagram.png)

Данная диаграмма отображает архитектуру системы регистрации, учета и устранения дефектов на автомобильном заводе.  
Компоненты системы разделены на **Client UI Core**, **Backend** и **Auxiliary Services**.

---

### Client UI Core
Объединяет интерфейсы взаимодействия с пользователем для четырех ролей:

- **Worker UI** — используется ремонтником для регистрации дефектов и выполнения ремонта.  
- **Dispatcher UI** — обеспечивает диспетчеру контроль свободных ремонтных мест и распределение автомобилей.  
- **Foreman UI** — используется бригадиром для назначения ремонтников.  
- **Quality Service UI** — интерфейс отдела контроля качества.

---

### Backend
Компонент, объединяющий логику всех подсистем, определяющих работу интерфейсов взаимодействия с пользователями.

- **External API Interface** — шлюз, принимающий запросы от UI и перенаправляющий их во внутренние сервисы.  
- **Notification Delivery and Registration Service** — отвечает за уведомления и регистрацию событий.  
- **Repair System** — включает компоненты **Repair Zone** и **Repair Place**, где моделируются ремонтные зоны и рабочие места; объединены в общую подсистему, отвечающую за ремонт.  
- **Reporting Service** — компонент, формирующий отчёты по сменам.  
- **DefectSystem** — регистрирует дефекты и хранит их описание, статус работы над ними.  
- **Control Core** — контролирует взаимодействие всех сервисов и координирует передачу данных между ними.

---

### Auxiliary Services
Компонент системы, объединяющий вспомогательные сервисы для бэкенда.  
Обеспечивает высокую надежность хранения и передачи данных.

## 3. [Диаграмма развёртки](./DeploymentDiagram.png)
![Диаграмма развёртки](./DeploymentDiagram.png)

На диаграмме представлены компоненты развёртки системы учёте дефектов.

Основные компоненты:
- Сервер приложений с REST API - является единой точкой обработки запросов от всех клиентов.
- База данных PostgreSQL - хранит данные о ремонтах, сменах, бригадах, а также различные метрики, необходимые для формирования отчётов.
- Брокер сообщений Kafka - управляет ремонтными очередями и отвечает за выдачи отчётов.
- Клиентские приложения:
    - Ремонтная зона: Регистрация рабочих, диагностика, назначение заданий.
    - Диспетчерская: Мониторинг ресурсов (зоны, рабочие).
    - Отдел контроля качества: Инициация ремонтного процесса.
